# PoC NL2SQL Framework - LLM 증축 계획
> 목표: 기존 멀티턴 에이전트 골격은 유지한 채, LLM으로 대체할 판단 경계면을 단계적으로 정리한다.
>
> 작성일 기준 실행 환경: `scale` 시나리오, `scale_augmented.jsonl`, 캐시 enabled.

## 왜 LLM 증축이 필요한가?

NL2SQL 서비스는 이미 멀티턴 에이전트의 핵심 골격을 갖추고 있다.  
thread_id 유지, follow-up 병합, pending 처리 흐름이 구현되어 있어, 기본적인 대화 오케스트레이션 자체는 충분히 동작한다.

그렇다면 LLM이 필요한 지점은 어디인가. 핵심은 대화 흐름 관리가 아니라 자연어 해석의 유연성이다. 예를 들어, 사용자의 표현은 동의어, 띄어쓰기 차이, 약어, 오탈자, 생략형 문장처럼 계속 변형된다. 또한 “그것”, “마지막 거”, “이거/그거”, “좀 더 자세히” 같은 지시어·참조어 기반 후속 질의는 규칙만으로 의도를 안정적으로 해석하기 어렵다. 여기에 스키마 변경이나 신규 용어 추가까지 발생하면, 기존 규칙 기반 해석은 쉽게 이탈한다.

문제는 규칙 기반 접근이 이런 표현 다양성을 모두 흡수하려면 지속적인 예외 처리와 규칙 추가가 필요하다는 점이다. 즉, 해석 범위를 넓힐수록 유지보수 비용이 빠르게 증가한다. 반면 LLM은 다양한 표현과 문맥을 일반화해서 처리할 수 있으므로, 이러한 해석 부담을 크게 줄여준다. 다만 그 대가로 지연 시간과 비용이 증가한다.

따라서, 이미 안정적으로 동작하는 멀티턴 오케스트레이션은 유지하고, 규칙 기반으로 확장 비용이 큰 “판단 계층”만 LLM으로 보강하거나 교체하는 방향이 더 적절하다.

---

## 1. 현재 시스템에서 LLM로 대체할 후보 정리

멀티턴 상태관리 엔진은 기존에 존재하므로 이번 증축 대상은 아래 `판단/해석` 지점이다.

- Query Preparation (`query_preparation`)
- Intent 분류 (`classify_intent`)
- NL2SQL 라우팅 (clarification 필요 여부 판단)
- SQL plan 템플릿 구성

이 중 특히 LLM로 바꾸면 효과가 큰 순서로 정리하면 다음과 같다.

### 1-1. 1순위: Query Preparation

현재는 템플릿에 가까운 규칙으로만 동작한다
- 장점: 빠름, 안정적
- 한계: 미완성 표현·동의어·문장 변형 처리 한계

**LLM 배치 이유**
- 정규화·보강(예: `다운타임 많은 장비 보여줘` → `다운타임 장비 top10` 같은 정규화)
- 잘못된 토큰 분리, 한글 띄어쓰기/약어, 동의어 처리

### 1-2. 2순위: Intent Router

의도 분류는 기존에도 휴리스틱 규칙으로 처리됨

**LLM 배치 이유**
- 규칙으로 처리하기 어려운 다의성 문장 처리
- 다중 의도(질의 + 조회 + 설명 요청) 동시 감지
- 이전 턴 히스토리 반영

### 1-3. 3순위: NL2SQL Clarification 라우팅

예: `장비 top10`만 들어왔을 때 지표가 비어 있으면 질문 유도

**LLM 배치 이유**
- 사용자 상태/히스토리에 맞는 질문 문구 생성
- 보류/재요청 정책의 일관성 확보

### 1-4. 4순위: SQL 생성 단계 일부

템플릿 조합 기반에서 벗어나, LLM이 최종 SQL plan 후보를 제시

**LLM 배치 이유**
- 쿼리 의도 해석과 DB 스키마 맵핑 강건성
- 단, `실행 보안`은 유지되어야 함(화이트리스트/파서/최종 검증 필요)

---

## 2. LLM로 바꾸지 않을 부분(고정 유지)

LLM 증축을 하더라도 아래는 유지하는 게 안전하다.

- SQLite 실행 엔진, SQL 실행 정책
- DB 스키마/권한/검증 guardrail
- Audit/요청 추적, request_id, metrics 수집
- 캐시 레이어(응답 재사용)
- 멀티턴 체크포인팅, thread/thread state 구조

이렇게 하면 “해석 실패”의 영향권을 좁히고, 핵심 인프라는 deterministic하게 남길 수 있다.

---

## 3. 단계별 마이그레이션 전략

### Phase 0: 인터페이스 고정(동작 동일)
- LLM 호출 지점마다 기존 output 형태와 동일한 스키마(JSON)로 인터페이스를 감쌈
- 기능 플래그로 `fallback` 가능

### Phase 1: Query Preparation만 LLM 전환
- 기존 룰 동시 운용 (`shadow`/`dual write`)
- 정규화 결과 품질 지표 비교

### Phase 2: Intent/Router 분리 전환
- 이미 있는 multi-turn `thread_id` + `messages` 상태를 활용해 맥락-aware 분기 강화
- clarification miss/over-question 비율을 관찰

### Phase 3: SQL plan 생성 보강
- 템플릿 + LLM 제안 하이브리드
- 실행 단계에서 strict parser/검증으로 안전성 확보

### Phase 4: 운영 안정화
- 성능 임계치(`p95`, `timeout`)에 대해 단계별 게이트 운영
- 실패율이 특정 임계치 위로 올라가면 즉시 heuristic 우선 모드로 롤백

---

## 4. 이 포스트의 전제

- 실험 기준은 `scale` 환경의 단일 스위트(참고: `scale_augmented.jsonl`)
- 휴리스틱 대비 LLM 증축의 핵심 목적은 **해석 범위 확장**, **서비스 안정성 유지**
