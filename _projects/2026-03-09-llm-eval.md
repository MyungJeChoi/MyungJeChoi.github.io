# PoC NL2SQL Framework - LLM 증축 성능 평가

> 같은 시나리오(`scale`)로 재현한 실험에서 휴리스틱 vs LLM 기반을 비교했다.

## 1. 비교 조건

- 비교 대상 1: `heuristic 기반` run `20260301_180027_072d8e480f`
- 비교 대상 2: `LLM 기반` run `20260309_211736_95c6dd24b2`
- 공통:
  - 시나리오: `scale`
  - 스위트: `scale_augmented.jsonl`
  - 캐시: `enable_cache = true`

---

## 2. 실험 결과

| 항목 | heuristic | LLM |
|---|---:|---:|
| passed | 51 | 51 |
| failed | 0 | 0 |
| pass rate | 100% | 100% |
| duration_s | 2.3507 | 53.3099 |
| latency p50 (ms) | 29.7778 | 740.8058 |
| latency p95 (ms) | 73.3796 | 885.5832 |
| nl2sql_requests_total | 43 | 42 |
| nl2sql_cache_hit_total | 4 | 5 |
| cache hit ratio | 9.30% | 11.90% |

- 정합성(pass)은 두 방식 모두 동일 100%로 같음.
- 하지만 지연시간은 크게 차이남.
  - p50 기준: `740.8058 / 29.7778 ≈ 24.9배`
  - p95 기준: `885.5832 / 73.3796 ≈ 12.1배`
  - 전체 duration: `53.3099 / 2.3507 ≈ 22.7배`
- 캐시 히트는 양쪽 모두 낮아(`5회 미만`) 성능 격차가 캐시 효과 때문이라고 보기는 어렵다.
- 초기 LLM 문제 버전(비교용 `20260309_210732_fae6d49623`)에서는 failed가 4건(총 51개 기준) 발생하여 현재보다 안정성 면에서 열악했고,  
  실패는 ETCH 계열 자산 정규화 편차에서 기인한 것으로 판단됨.
  - 이 경우, LLM 정규화 prompt을 강화하여 인식을 강화하였고, 그 결과 pass rate 100%을 기록함

---

## 3. 운영 관점 판단

결론적으로 현재 데이터로는 실서비스 기본 채택에선 다음이 안전하다.

- 단기: heuristic 경로를 기본, LLM은 `preparation`/`router` 보조로 부분 적용
- 중기: LLM 경로는 캐시 적중률이 올라가는 대화 패턴에서만 비용 회수 시도
- 장기: 비용 모델(토큰/초당 처리량/동시성)까지 넣어 라우팅 규칙을 동적으로 튜닝

---

## 4. 다음 예정 실험

1. 워크로드별 분해: 단발 query / follow-up / clarification 비율별 분리 비교
2. 캐시 강화 실험: 동일 자산·동일 지표 반복 질의 비중 조정
3. 회귀 기준: p95 latency 상한치(예: 1000ms) 및 실패율 상한치(예: 1%)를 넘을 때 자동 rollback
4. 비용 지표 결합: 토큰 사용량 + 처리량 + 사용자 만족도(clarification 완료율)

> LLM이 “맞춤형 해석”을 줬지만, 현재 실험에서는 정확도보다 비용과 지연에서 손해가 크다.
> 다음 단계는 **성능-품질 트레이드오프를 수치로 통제할 수 있는 라우팅 전략**을 붙이는 것이다.
